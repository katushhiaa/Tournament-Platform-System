# Специфікація потоків даних та інтерфейсів взаємодії (Data Flow & Interaction Specification)

Даний документ формалізує архітектурні принципи маршрутизації інформаційних потоків у межах програмного комплексу "Tournament Platform System". Специфікація визначає топологію високорівневої взаємодії компонентів системи, деталізує сценарії обробки ключових транзакцій та регламентує технічні механізми транспортування і валідації даних.

## 1. Високорівнева схема архітектурної взаємодії (High-Level Architecture)

Інформаційна взаємодія в системі базується на класичній трирівневій клієнт-серверній архітектурі, що забезпечує чітке розмежування відповідальності між репрезентативним, логічним та персистентним шарами.

![Структурна модель взаємодії(Діаграма)](https://github.com/user-attachments/assets/3d42dc37-2853-4767-ad66-0498e7d081f7)


### Опис компонентів:
* **Клієнтський рівень (Frontend):** Ініціює HTTP-запити до серверної частини, відповідає за збір вхідних даних від користувача, їх первинну клієнтську валідацію та рендеринг отриманих відповідей.
* **Рівень застосунку (Backend):** Виступає єдиною точкою входу для клієнтських запитів. Здійснює маршрутизацію, перевірку прав доступу (RBAC), валідацію вхідних даних (через DTO), виконання бізнес-логіки та формування стандартизованих відповідей.
* **Рівень даних (Data Layer):** Забезпечує довгострокове зберігання (персистентність) інформації, гарантуючи цілісність даних при виконанні транзакцій.

## 2. Сценарії взаємодії та шляхи передачі даних (Data Flow)

Нижче наведено технічний шлях обробки інформації (Request/Response Cycle) для трьох фундаментальних бізнес-процесів системи.

### Сценарій 2.1: Процедура створення нового турніру
*Цільовий процес: Ініціалізація змагання авторизованим користувачем з правами "Організатор".*

1.  **Request (Клієнт):** Відправлення POST-запиту на ендпоінт `/api/v1/tournaments`. Тіло запиту містить необхідні параметри (назва, дати, формат). У заголовку `Authorization` передається JWT-токен.
2.  **Authentication & Authorization (Сервер):** Middleware перевіряє валідність підпису JWT-токена та наявність у користувача ролі "Organizer".
3.  **Validation (Сервер):** Вхідний JSON-об'єкт трансформується у `CreateTournamentDTO`. Здійснюється перевірка відповідності типів даних та бізнес-правил (наприклад, дата початку не може бути в минулому).
4.  **Database Transaction (Сервер -> БД):** У разі успішної валідації генерується унікальний ідентифікатор (UUID), і формується SQL-запит (INSERT) для запису даних у таблицю `tournaments`.
5.  **Response (Сервер -> Клієнт):** База даних повертає підтвердження збереження. Сервер формує HTTP-відповідь зі статусом `201 Created` та повертає об'єкт новоствореного турніру у форматі JSON.

### Сценарій 2.2: Процедура реєстрації учасника на турнір
*Цільовий процес: Приєднання користувача з роллю "Гравець" до існуючого турніру зі статусом "draft" або "registration_open".*

1.  **Request (Клієнт):** Відправлення POST-запиту на ендпоінт `/api/v1/tournaments/{id}/participants` із зазначенням ідентифікатора турніру та JWT-токеном гравця.
2.  **Validation & Business Logic (Сервер):** * Перевірка існування турніру за вказаним ідентифікатором.
    * Перевірка статусу турніру (чи дозволена реєстрація).
    * Перевірка ліміту учасників (чи не досягнуто `max_participants`).
    * Перевірка на відсутність дублювання (чи не зареєстрований даний гравець раніше).
3.  **Database Transaction (Сервер -> БД):** Додавання запису до зв'язкової таблиці `tournament_participants` із зазначенням `user_id` та `tournament_id`.
4.  **Response (Сервер -> Клієнт):** Повернення статусу `201 Created` із підтвердженням успішної реєстрації.

### Сценарій 2.3: Процедура генерації турнірної сітки та механізм BYE
*Цільовий процес: Перехід турніру в статус "active" та формування сітки Single Elimination.*

1.  **Request (Клієнт):** POST-запит на `/api/v1/tournaments/{id}/start` із зазначенням ідентифікатора турніру та JWT-токеном організатора.
2.  **Algorithm (Сервер):** Визначається кількість учасників $N$.
    * Якщо $N$ не є ступенем двійки ($2^k$), алгоритм розраховує кількість технічних перемог (BYE): $Matches\_to\_skip = 2^k - N$, де $2^k$ — найближчий більший ступінь двійки.
    * Учасники з BYE автоматично отримують заповнене поле `winner_id` у першому раунді.
3.  **Database Transaction:** **Bulk Insert** матчів першого та наступних раундів.
    * Для матчів із BYE встановлюється `is_valid = true`, а переможець автоматично прописується в `team_a_id` або `team_b_id` матчу наступного рівня (`level - 1`).
4.  **Response:** Повернення `200 OK` із повною структурою згенерованої сітки.

## 3. Механізми передачі та збереження стану даних

### 3.1. Стратегія автентифікації: Access & Refresh Tokens
Для забезпечення безпеки (NFR Security) та зручності користувача (UX) система використовує двоступеневу модель JWT:

| Тип токена | Термін дії (TTL) | Опис |
| :--- | :--- | :--- |
| **Access Token** | **15 хвилин** | Короткоживучий токен для авторизації кожного запиту. Передається в заголовку `Authorization: Bearer`. |
| **Refresh Token** | **7 днів** | Довгоживучий токен для отримання нового Access Token без повторного введення пароля. Зберігається в `HttpOnly Cookie` для захисту від XSS. |

**Механізм оновлення:** При отриманні помилки `401 Unauthorized` (через вихід TTL Access Token), клієнт автоматично звертається до ендпоінту `/api/v1/auth/refresh` з Refresh токеном.

### 3.2. Використання об'єктів передачі даних (Data Transfer Objects - DTO)
Для забезпечення суворої типізації, інкапсуляції внутрішньої логіки та запобігання вразливостям (наприклад, Mass Assignment Vulnerability), передача даних між клієнтським та серверним рівнями суворо регламентована використанням DTO.

* **Вхідні DTO (Input):** Використовуються для перехоплення HTTP-запитів. Перед передачею даних до шару бізнес-логіки об'єкти проходять автоматичну валідацію (наприклад, перевірка довжини рядків, форматів email, допустимих значень enum). Будь-яка невідповідність призводить до негайного відхилення запиту з помилкою `400 Bad Request`.
* **Вихідні DTO (Output/Response):** Застосовуються для форматування відповідей бази даних перед відправкою клієнту. Це гарантує вилучення конфіденційної інформації (наприклад, гешів паролів, системних метаданих) з фінального JSON-пакета.

### 3.3. Global Exception Handling (Обробка помилок)
Система використовує єдиний стандарт повернення помилок, що дозволяє Frontend-частині консистентно відображати сповіщення користувачу.

**Структура об'єкта помилки (Error DTO):**
```json
{
  "timestamp": "2026-04-06T13:11:21Z",
  "status": 400,
  "error": "Bad Request",
  "message": "Validation failed",
  "details": {
    "start_date": "Дата початку не може бути в минулому",
    "max_teams": "Мінімальна кількість учасників для Single Elimination — 2"
  },
  "path": "/api/v1/tournaments"
}
```

* **400 Bad Request:** Помилки валідації вхідних DTO.
* **401 Unauthorized:** Відсутній або прострочений Access токен.
* **403 Forbidden:** Недостатньо прав (наприклад, гравець намагається видалити турнір).
* **404 Not Found:** Ресурс (турнір/матч) не існує.
* **422 Unprocessable Entity:** Порушення бізнес-логіки (наприклад, спроба почати турнір з 1 учасником).

### 3.5. Парадигма відсутності стану (Statelessness)
Архітектура прикладного програмного інтерфейсу побудована на фундаментальному принципі REST – відсутності збереження стану сесії на сервері (Stateless Architecture).

* **Локалізація стану:** Серверний простір пам'яті не використовується для зберігання інформації про поточні сесії користувачів.
* **Механізм автентифікації:** Кожен HTTP-запит функціонує ізольовано та є самодостатнім. Ідентифікація суб'єкта здійснюється виключно шляхом розшифрування криптографічного JSON Web Token (JWT), який передається в кожному запиті.
* **Переваги підходу:** Дана модель мінімізує використання оперативної пам'яті сервера, виключає необхідність синхронізації сесій та забезпечує високий рівень горизонтальної масштабованості системи.


Ось оновлена версія специфікації з урахуванням усіх погоджених архітектурних правок: механізму BYE, системи обробки винятків та вдосконаленої стратегії автентифікації.
