## 1. Introduction

Цей документ визначає стратегію, обсяг, методи та критерії завершення тестування MVP платформи для організації та проведення спортивних турнірів. Мета — забезпечити перевірку відповідності системи бізнес-вимогам та гарантувати стабільну роботу всіх ключових функцій перед фінальною демонстрацією.

**Об'єкт тестування:** веб-застосунок Tournament Platform (Vue.js 3 Frontend + ASP.NET Core MVC REST API + PostgreSQL).

**Відповідальні за тестування:** уся команда відповідно до RACI-матриці.

## 3. Testing Strategy & Approach

### 3.1 Рівні тестування

**Unit Testing**
- Мета: перевірка ізольованої бізнес-логіки (алгоритм генерації сітки, визначення переможця, Bye-слоти).
- Що тестуємо: функція генерації пар Single Elimination, логіка підрахунку Bye-слотів, валідація рахунку матчу.
- Інструменти: **xUnit** (C# бекенд), **Vitest** (Vue.js 3 + TypeScript frontend).

**Integration Testing**
- Мета: перевірка взаємодії Frontend ↔ API ↔ БД.
- Що тестуємо: POST /auth/register → запис у БД; POST /tournament → статус Draft у БД; POST /match/result → оновлення сітки.
- Інструменти: Postman (Collections + Automated Tests), Swagger UI.

**System / Manual E2E Testing**
- Мета: наскрізна перевірка повних сценаріїв користувача.
- Сценарії: «Організатор реєструється → створює турнір → додає гравців → запускає → вводить результати → турнір завершено»; «Гравець реєструється → знаходить турнір → подає заявку → бачить сітку».
- Інструменти: ручне тестування у браузері Chrome/Firefox, Browser DevTools.

### 3.2 Види тестування

| Вид | Що перевіряємо | Метод |
|-----|----------------|-------|
| **Functional** | Відповідність 29 UC специфікації | Manual + API |
| **Negative / Boundary Value** | Граничні значення, некоректні дані, бізнес-логіка | Manual + Postman |
| **Security (базовий)** | Авторизація ендпоінтів (JWT), ролі ORGANIZER/PLAYER, SQL Injection у полях введення | Postman + Manual |
| **UI/UX** | Адаптивність, стан кнопок (disabled/active), toast-повідомлення, loading states | Manual (Chrome DevTools) |
| **API** | HTTP-коди (200/201/400/401/403/404/409/500), структура JSON-відповідей | Postman + Swagger UI |

## 5. Entry Criteria (коли можна починати тестування)

Тестування розпочинається лише за виконання всіх умов:

- [ ] Код змерджено у гілку `develop` або `main`
- [ ] Застосунок розгорнуто на тестовому середовищі (локально або staging)
- [ ] База даних розгорнута та заповнена тестовими даними (мінімум: 3 тестових акаунти з ролями ORGANIZER та PLAYER, 2 тестових турніри)
- [ ] API-документація (Swagger UI / Postman Collection) актуальна та доступна
- [ ] Виправлено блокуючі баги попереднього спринту

## 6. Exit Criteria (коли тестування вважається завершеним)

### 6.1 Критерії за тест-кейсами

| Критерій | Порогове значення |
|----------|------------------|
| Позитивні тест-кейси зі статусом **Pass** | **100%** (58/58) |
| Негативні тест-кейси зі статусом **Pass** | **≥ 90%** (≥ 27/29 — допускається 2 low-priority Fail) |
| Тест-кейси модулів **Critical** пріоритету зі статусом Pass | **100%** без виключень |

### 6.2 Критерії за дефектами (Bug-free умови)

| Severity | Умова |
|----------|-------|
| **Critical** (застосунок не запускається, втрата даних, помилка генерації сітки) | **0 відкритих** — усі виправлені та перевірені |
| **High** (основний функціонал не працює: неможливо зареєструватись, ввести результат) | **0 відкритих** — усі виправлені |
| **Medium** (другорядний функціонал, UI-артефакти) | **≤ 3 відкритих** з прийнятим workaround |
| **Low** (косметичні помилки, орфографія) | Фіксуються, виправлення необов'язкове до демонстрації |

### 6.3 Функціональні критерії (обов'язково Pass)

Наступні наскрізні сценарії мають виконуватися без будь-яких помилок:

1. **Реєстрація та вхід:** Новий користувач може зареєструватись з роллю ORGANIZER або PLAYER та увійти в систему → JWT-токен видається → редирект коректний.

2. **Повний цикл турніру:** Організатор створює турнір → додає 8 учасників → система генерує сітку → організатор вводить результати → турнір завершується з визначеним чемпіоном.

3. **Реєстрація гравця:** Авторизований гравець знаходить турнір → подає заявку → отримує підтвердження → з'являється у списку учасників.

4. **Захист від некоректних дій:** Система не повертає HTTP 500 у жодному з 29 негативних тест-кейсів — лише коректні повідомлення про помилки (400/401/403/404/409).

5. **Розмежування ролей:** Гравець не може створити турнір або ввести результат матчу. Неавторизований відвідувач не має доступу до захищених ендпоінтів.

### 6.4 API-критерії (Postman + Swagger UI)

- Усі ендпоінти повертають коректні HTTP-статус-коди відповідно до специфікації
- Структура JSON-відповідей відповідає очікуваному формату (наявність обов'язкових полів: `id`, `status`, `data`)
- Захищені ендпоінти повертають **HTTP 401** без токена та **HTTP 403** при недостатніх правах

### 6.5 UI/UX-критерії

- Відсутні видимі помилки у консолі браузера (JavaScript errors) під час проходження основних сценаріїв
- Loading-стани (Spinner) відображаються під час усіх асинхронних запитів
- Toast-повідомлення (success/error) з'являються для всіх ключових операцій
- Кнопки коректно переходять у стан `disabled` після дії (запобігання подвійному надсиланню)

## 8. Deliverables (Очікувані результати)

- **Test Case Matrix** — `Test Cases.docx` (58 тест-кейсів для 29 UC, статуси Pass/Fail)

## 9. Risks & Mitigations

| Ризик | Ймовірність | Дія |
|-------|-------------|-----|
| API-ендпоінти не готові до початку тестування | Середня | Тестування UI незалежно; мокування API через Postman Mock Server або Swagger UI |
| Алгоритм генерації сітки дає неочікувані результати при непарній кількості | Висока | Пріоритизувати xUnit Unit-тести UC-13, UC-14 на Етапі 1 |
| Нестача часу на regression після виправлення багів | Середня | Обмежити regression тільки Critical та High тест-кейсами |
| Відсутність тестових даних у БД | Низька | Підготувати seed-скрипт або SQL-дамп із тестовими акаунтами (EF Core Data Seeding) |
